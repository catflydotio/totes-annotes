Remote code-server speedrun

The repository contains three files: Dockerfile, entrypoint.sh, and settings.json.

Dockerfile

We'll use a multi-stage build; the first stage lets us get the code for tired-proxy from its public Docker image.

FROM lubien/tired-proxy:2 as proxy

We're planning to do Elixir development in our remote environment, so for the second stage we choose an official Elixir image that provides elixir, mix, etc., leaving us less to configure.

FROM hexpm/elixir:1.12.3-erlang-24.1.4-debian-bullseye-20210902-slim

Do updates and get remaining dependencies. Unzip and curl are needed to install code-server.

# install build dependencies
RUN apt-get update -y \
    && apt-get install -y build-essential git unzip curl \

Clear out the apt cache and purge apt lists, since we won't need these in the image.

    && apt-get clean && rm -f /var/lib/apt/lists/*_* \

Install code-server.

    && curl -fsSL https://code-server.dev/install.sh | sh
# prepare build dir
WORKDIR /app

Set the default shell to Bash.

# Use bash shell
ENV SHELL=/bin/bash

We're supplying a settings.json file for Code Server, mainly for the dark theme! Copy this from the local working directory.

# Apply VS Code settings
COPY settings.json /root/.local/share/code-server/User/settings.json

We want to do a few things once the machine starts up, so we put all that into a shell script, entrypoint.sh. Copy that in too.

# Use our custom entrypoint script first
COPY entrypoint.sh /entrypoint.sh

Install flyctl, the Fly.io CLI tool, and add it to our PATH. This lets us deploy apps from the code-server terminal.

RUN curl -L https://fly.io/install.sh | sh \
    && echo 'export FLYCTL_INSTALL="/root/.fly"' >> ~/.bashrc \
    && echo 'export PATH="$FLYCTL_INSTALL/bin:$PATH"' >> ~/.bashrc \

Install the Elixir Language Server extension for Code Server.

    && code-server --install-extension elixir-lsp.elixir-ls

Copy over the executable for the Tired Proxy from the first stage.

COPY --from=proxy /tired-proxy /tired-proxy

Run entrypoint.sh whenever the VM starts up.

ENTRYPOINT ["/entrypoint.sh"]
entrypoint.sh

Set the TIME_TO_SHUTDOWN environment variable to 3600 seconds (1 hour). This is used in the tired-proxy command below.

#!/bin/bash

TIME_TO_SHUTDOWN=3600

Create a folder for the Elixir project to live in. The -p tag prevents errors in the case that /project already exists (as it should the second time you start the VM).

mkdir -p /project

fly volumes generates a lost+found folder. Remove it so we can check for other files.

# In case fly volumes put something there
rm -rf '/project/lost+found'

If the /project dir is now empty, we want to initialize it.

if [ -z "$(ls -A /project)" ]; then
    echo "Preparing project"

Make a fresh clone of the git repository of the project we want to work on. (We'll set the GIT_REPO environment variable when we run the VM.)

    rm -rf /project
    git clone $GIT_REPO /project

    cd /project

Install hex and rebar locally (non-interactively with --force), and get project dependencies.

    echo "Setting up Elixir environment"

    mix local.hex --force
    mix local.rebar --force
    mix deps.get
fi

This is the best part of the script: We spawn a Code Server, with the /project folder open, listening on port 9090â€”but we won't expose this port directly. Tired Proxy just maps port 8080 to 9090, and if there's no incoming HTTP connection for $TIME_TO_SHUTDOWN seconds, it exits. That's it.

code-server --bind-addr 0.0.0.0:9090 /project &
    /tired-proxy --port 8080 --host http://localhost:9090 --time $TIME_TO_SHUTDOWN

In the local terminal

create a new app on Fly.io with a name and an organization. The VM we'll start and stop later will be tied to this app.

The remote IDE URL will be <your-app-name>.fly.dev.

$ fly apps create <your-app-name>

Create a new volume called storage, tied to this app, with size 2GB. You can choose to make it smaller.

$ fly volumes -a <your-app-name> create storage --size 2

You will be prompted to select a region. Choose one nearish you for lowest latency. The VM will be tied to the hardware this volume is on. In the example we've chosen gru.

By default, Fly.io apps have private IPV6 addresses for use within their organization's WireGuard network. If we want to access this app without a WireGuard tunnel, we need to give it a public IP.

The public IP for a standard (non-Machines) Fly.io App gets allocated during the deploy process, but we're not running fly deploy on this app, so we need to allocate this manually.

$ fly ips allocate-v4 -a <your-app-name>

We use fly secrets to pass secret environment variables into a Fly.io app.

$ fly secrets set -a <your-app-name> \

code-server asks for a password when you first open it. It will generate its own random password on installation if we don't set the PASSWORD environment variable. We can still ssh into the VM later and extract it from a config file; but it's easier to set it ahead.

  PASSWORD=<mypassword> \

We give the app our fly auth token. so that it can deploy our project app to Fly.io.

  FLY_API_TOKEN=$(fly auth token)
Secrets are staged for the first deployment

Run a new Fly Machines VM. The first argument is the image or the path to the Dockerfile. In this case it's the current folder.

$ fly machine run . \

Map port 8080 to the external HTTPS port (443) so we can access it via your-app-name.fly.dev. Remember this port belongs to the Tired Proxy so the server shuts down if no one uses it for a while.

  -p 443:8080/tcp:tls \

Elixir Phoenix uses port 4000 for development. Open this port so we can see our dev server on <your-app-name>.fly.dev:4000. Note: This does mean that the world can see it too.

  -p 4000:4000/tcp:tls \

For the best experience (i.e. to avoid OOM crashes), we'll go for the recommended mimimum of 1GB. You can try a lower value, but keep this choice in mind if you have to debug a VM crash! 

  --memory 1024 \

In order to mount the volume we provisioned earlier, the three-letter code <your-region> must match the volume's region.

  --region <your-region> \

Mount the volume called storage to the path /project. That's where our project files will live.

  --volume storage:/project \

This machine is already on your organization's WireGuard private network, so it doesn't need to create a userland WireGuard tunnel to reach a remote builder. We use the FLY_REMOTE_BUILDER_HOST_WG environment variable to tell flyctl to use native WireGuard, which means it'll be faster to deploy apps from our Code Server.

  --env FLY_REMOTE_BUILDER_HOST_WG=1 \

Store the URL for the repo to be cloned to /project. We're using a prepared Phoenix / SQLite example for fun (and so we can focus on deployment here, and not databases).

  --env GIT_REPO=https://github.com/fly-apps/hello_elixir_sqlite.git \

Tell flyctl to deploy the VM under the dev server app.

  -a <your-app-name>

It's time to code

Now we can visit our freshly-minted remote development environment at <your-app-name>.fly.dev. It'll ask for the password we set earlier with fly secrets set. Fear not the light theme. Once our settings.json is read in, it'll switch to dark.

Once we're in, we have the /project folder open, complete with the cloned hello_elixir_sqlite project files. README.md contains instructions for building, previewing, and deploying that app. In case you're just following along in your imagination, we'll repeat them here.

Open the integrated VS Code terminal with ^+`.

In the remote machine's terminal

Compile the app and start up the dev server.

Visit it in the browser at <your-app-name>.fly.dev:4000.

$ mix phx.server

In our VM's Dockerfile, we installed flyctl, so we can run any fly command from the integrated terminal. We're authed, because we set the FLY_API_TOKEN secret, which our CLI will read from the environment if it's available.
